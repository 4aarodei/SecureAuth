# Технічне завдання

- Рівень: Strong Junior C# / ASP.NET Core

## Опис задачі

Необхідно реалізувати контролер з трьома `POST`-методами, що забезпечують безпечний вхід у систему.

- Токени мають бути `opaque` рядками, не `JWT`
- Кожен запит обов'язково підписується за допомогою `ApiSignature`
- Використання зовнішніх бібліотек для автентифікації заборонено

## 1. Алгоритм підпису (`ApiSignature`)

Підпис запиту обчислюється наступним чином:

```text
ApiSignature = SHA-256(StaticKey + RequestDate)
```

Де:

- `StaticKey` - секретний рядок з конфігурації, наприклад `appsettings.json` або `env`-змінна; він відомий і клієнту, і серверу
- `RequestDate` - Unix timestamp у мілісекундах: `DateTimeOffset.UtcNow.ToUnixTimeMilliseconds()`

Логіка валідації на сервері:

1. Сервер отримує `requestDate` з тіла запиту.
2. Незалежно обчислює підпис.
3. Порівнює його з надісланим `apiSignature`.
4. Перевіряє "свіжість" (`freshness`): різниця між `requestDate` і поточним серверним часом не повинна перевищувати `±5 хвилин`.

Якщо підпис невірний або запит застарів, сервер має повернути `401 Unauthorized`.

## 2. Методи контролера

### `POST /auth/login`

Приймає:

- `login`
- `password`
- `apiSignature`
- `requestDate`

Повинен виконати:

1. Валідацію підпису та `freshness`
2. Перевірку `credentials`:
   пароль порівнюється з хешем у сховищі

Результат:

- генерує `simpleToken` з `TTL 5 хв`
- зберігає його у сховищі
- повертає клієнту

### `POST /auth/token`

Приймає:

- `simpleToken`
- `apiSignature`
- `requestDate`

Повинен виконати:

1. Валідацію підпису та `freshness`
2. Перевірку `simpleToken`:
   токен має існувати в сховищі та бути актуальним

Важливо:

- `simpleToken` є одноразовим
- після успішної перевірки він має бути негайно видалений

Результат:

- генерує `fullToken` з `TTL 24 год`
- повертає його клієнту

### `POST /auth/logout`

Приймає:

- `fullToken`
- `apiSignature`
- `requestDate`

Повинен виконати:

1. Валідацію підпису та `freshness`
2. Видалення `fullToken` зі сховища

Результат:

- `200 OK`
- якщо токен не знайдено, повертається `401 Unauthorized`

## 3. Технічні вимоги

- Порядок перевірки:
  підпис і `freshness` перевіряються першими, до будь-якої бізнес-логіки
- Безпека паролів:
  у сховищі зберігаються лише хеші, жодного `plain-text`
- Генерація токенів:
  використовувати `RandomNumberGenerator` для створення криптографічно стійких рядків, не `Guid.NewGuid()`
- Сховище:
  реалізувати `thread-safe` сховище, наприклад через `ConcurrentDictionary`, з реєстрацією в `DI` як `Singleton`
- Формат помилок:
  однотипний `JSON` для всіх помилок

```json
{
  "error": "...",
  "message": "..."
}
```

## 4. Буде плюсом

- Винесення перевірки підпису в `ActionFilter`, щоб не дублювати код у методах
- Реалізація `IHostedService` для фонового очищення протухлих токенів
- Коротке пояснення:
  чому використання `requestDate` без перевірки `freshness` не захищає від `replay`-атак, тобто повторного використання перехопленого запиту
